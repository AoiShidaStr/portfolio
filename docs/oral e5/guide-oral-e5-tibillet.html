Les 6 étapes de mise en service

#ActionTechnologie
1Installer le système d'exploitationDebian 13, LVM, sudo
2Isoler les services dans des conteneursDocker, docker-compose

5 Partie 4 — C3 Développement de fonctionnalités 4 min

"En parallèle de la mise en service, j'ai développé plusieurs fonctionnalités demandées dans le cahier des charges, directement sur le code source Django de TiBillet."

Les 6 réalisations de développement

#277 Affichage conditionnel de l'heure d'un événement
Ajout du champ show_time = BooleanField dans BaseBillet/models.py → propagé dans l'admin Django (list_display + fields dans EventAdmin) → conditionnel dans 2 templates : partial/list.html (agenda) et retrieve.html (page détail). Nécessité de faire makemigrations + flush.sh pour appliquer la migration.
#257 Affichages configurables de la billetterie
Ajout de show_price_note (BooleanField) et price_note (CharField) dans models.py pour masquer ou personnaliser le bloc prix. Création de la balise template event_dates dans tibitags.py — calcule automatiquement l'affichage humain des dates (1 jour, même jour avec fin, multi-jours). Mis à jour dans 4 templates : cardEvent.html, cardCreateEvent.html, retrieve.html, booking.html.
#191 Affichage de la date et heure de fin d'événement
Modification de BaseBillet/templates/reunion/views/event/retrieve.html — ajout d'un bloc conditionnel {% if event.end_datetime %} avec gestion de deux cas : même jour (affiche seulement l'heure de fin) ou jours différents (affiche la date complète de fin).
#180 Description courte dans l'agenda
Modification de partial/list.html — ajout d'un bloc {% if event.short_description %} affichant la description sous le titre de l'événement dans la liste de l'agenda.
Frontend Filtre multi-critères des événements par date
Modification de BaseBillet/views.py : ajout du paramètre date dans federated_events_filter() et partial_list(). Modification de partial/list.html : les titres de dates deviennent des liens cliquables ?date=YYYY-MM-DD. Création de search2.html pour un panneau "Options avancées" (toggle CSS checkbox). Section advanced-search avec animation max-height.
Back-end Recherche multi-critères des billets dans l'admin
Modification de Administration/admin_tenant.py — ajout de 'first_name', 'last_name', 'reservation__user_commande__email' dans list_filter de la classe TicketAdmin, en complément des filtres existants.
Contenu Mentions légales du site de billetterie
Rédaction complète des mentions légales conformes LCEN (loi 2004-575) et RGPD : éditeur (Association Komidi, RNA W9R2002592), hébergeur (OVHcloud), solution technique (TiBillet AGPL-3.0, lauréat France 2030), propriété intellectuelle, traitement des données, paiement sécurisé, cookies.

Tu as travaillé sur models.py, views.py, admin.py et des templates Django — toute la pile MVC. Ce n'est pas de la configuration, c'est du développement back-end et front-end sur une vraie codebase open source.

6 Partie 5 — C2 Incidents résolus 3 min

Incident 1 — Bug VPS (production)

"Sur le VPS de production de Komidi, le site ne se chargeait plus. Django plantait dès la page d'accueil."

Diagnostic : un Client en base de données avait une mauvaise valeur de categorie pour son schema_name — Django ne pouvait plus résoudre le tenant et levait une exception non catchée.

Solution :

docker exec -it lespass_django sh -lc "
python manage.py shell -c \"
from Customers.models import Client
c = Client.objects.get(schema_name='ce13207cf99c412986c8d6fec1943d3d')
c.categorie = 'S'
c.save()
\""

Résultat : site opérationnel immédiatement après l'exécution de la commande.

Incident 2 — Environnement de dev instable (Docker)

Chaîne causale complète :

CAUSE RACINE (identifiée a posteriori) :
  LaBoutik était dans docker-compose.yml sans être nécessaire
↓
Effet 1 : surcharge mémoire/CPU
↓
Effet 2 : conteneurs qui redémarrent en boucle (restart loop)
↓
Effet 3 : ./flush.sh et rsp s'interrompent sans raison apparente
↓
Effet 4 : diagnostic difficile — vérifications manuelles (git branch, docker version, logs)
↓
SOLUTION : suppression LaBoutik dans docker-compose.yml
+
Synchronisation : git stash → git pull
↓
RÉSULTAT : environnement stable, tous les scripts fonctionnels ✓

Formule à utiliser devant le jury

"Le problème difficile n'était pas la correction — supprimer LaBoutik du docker-compose.yml était simple une fois la cause identifiée. La vraie difficulté était le diagnostic : j'ai dû croiser les logs Docker, la version Docker, et l'état des branches Git avant de comprendre que c'était la présence d'un service inutilisé qui saturait les ressources et déstabilisait l'ensemble."

8 Questions probables du jury

QuestionRéponse préparée
"C'est quoi Docker ?" Outil de conteneurisation : Fedow, Lespass et PostgreSQL tournent chacun dans un conteneur isolé. Avantage : reproductible sur n'importe quelle machine sans conflit de dépendances.
"Qu'est-ce que Django ?" Framework web Python qui suit le pattern MVC (Model-View-Template). TiBillet l'utilise pour gérer la base de données (models), la logique métier (views) et l'affichage (templates HTML).
"Pourquoi Fernet ?" Algorithme de chiffrement symétrique. TiBillet l'utilise pour sécuriser les communications entre Fedow et Lespass — chaque message est chiffré avec la même clé partagée dans les deux .env.
"C'est quoi un BooleanField Django ?" Un champ de modèle qui stocke True ou False en base de données. Pour show_time et show_price_note, il permet à l'admin Komidi d'activer/désactiver l'affichage sans modifier le code.
"Qu'est-ce qu'une migration Django ?" Quand on ajoute un champ au modèle, Django doit mettre à jour la structure de la base de données. makemigrations génère le fichier de migration, et flush.sh l'applique ici.
"C'est quoi un tenant Django ?" Isolation multi-tenant : public (utilisateurs partagés), lespass (données billetterie), meta (futur agrégateur). Permet à une instance TiBillet de gérer plusieurs organisations.
"Qu'est-ce que AGPL-3.0 ?" Licence copyleft — le code source doit rester libre et toute modification doit être redistribuée sous la même licence, même pour un service en ligne (ce qui renforce la philosophie coopérative de TiBillet).
"Vous avez contribué au code officiel de TiBillet ?" Non — j'ai travaillé sur le fork de Komidi. Ma mission était d'adapter TiBillet aux besoins spécifiques de l'association, pas de contribuer au projet upstream.
"C'est quoi le test-driven development dans le nom du repo ?" C'est le nom du repo de configuration Docker fourni par TiBillet pour l'environnement de développement local. Le nom reflète leur démarche de développement — pas un framework de test.
"Pourquoi VSCodium ?" Version open source de VS Code sans télémétrie Microsoft — cohérent avec la philosophie open source et coopérative de TiBillet et de Komidi.
"Comment avez-vous diagnostiqué le Bug VPS ?" Les logs Django montraient une exception au chargement de la page d'accueil liée au résolveur de tenant. En inspectant le modèle Client, j'ai trouvé que la categorie du schema_name était incorrecte — corrigée directement via manage.py shell.
"Les mentions légales, c'est du développement ?" C'est de la production de contenu avec une contrainte légale forte (LCEN + RGPD). Ça fait partie de la mise en ligne d'un service — compétence C3.

9 Commandes essentielles à mémoriser

# ── Gestion des conteneurs ─────────────────────────────
docker compose up -d         # démarre en arrière-plan (detached)
docker compose down          # arrête et supprime les conteneurs
docker compose logs -f       # suit les logs en temps réel (Ctrl+C pour quitter)
docker ps                    # liste les conteneurs actifs (statut "up")
docker network create frontend  # crée le réseau partagé entre services

# ── Accès aux conteneurs ───────────────────────────────
docker exec -it lespass_django bash   # entre dans lespass
docker exec -it fedow_django bash     # entre dans fedow
./flush.sh                             # initialise la base (fedow d'abord, lespass ensuite)
sp                                     # shell_plus Django (autocomplétion)

# ── Vérification finale ────────────────────────────────
config = Configuration.get_solo()
# → <Configuration: Settings for Lespass>  ✓
fedow.can_fedow()
# → True  ✓

# ── Django migrations ──────────────────────────────────
poetry run python manage.py makemigrations  # génère le fichier de migration
./flush.sh                                  # applique les migrations + réinitialise

# ── Git (résolution incidents) ─────────────────────────
git stash     # met de côté les modifs locales
git pull      # récupère les mises à jour distantes
git branch    # vérifie la branche active dans chaque repo

10 Ce qui te différencie des autres candidats

1. Quatre compétences E5 sur un seul stage
C2 + C3 + C4 + C5 — avec des preuves concrètes et documentées pour chacune. La plupart des candidats en couvrent 2 maximum par réalisation.
2. Vrai code open source — toute la pile Django
models.py, views.py, admin.py, templates, balises custom — tu as travaillé sur toutes les couches de la stack MVC d'un projet réel sous licence AGPL-3.0, lauréat France 2030.
3. Deux incidents documentés (dev + production)
Bug VPS corrigé en production via Django shell + 5 problèmes Docker résolus en développement. Chaîne causale reconstruite a posteriori — c'est du diagnostic méthodique, pas du tâtonnement.
4. Issues numérotées traçables
#180, #191, #257, #277 — chaque fonctionnalité est documentée dans un fichier .docx séparé avec le contexte, les fichiers modifiés, et les lignes approximatives. Niveau de documentation professionnel.
5. Livrable autonome de qualité professionnelle
Le guide HTML peut servir à n'importe quel contributeur sans toi. Section troubleshooting, blocs de code copiables, alertes contextuelles — c'est un document de production.

Entraîne-toi à chronomètrer. 15 minutes maximum pour l'exposé — le jury coupe à la sonnerie. Mieux vaut finir à 13 min et laisser 17 min aux questions qu'être interrompu.

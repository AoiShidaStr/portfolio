0 Posture globale — les 15 minutes
Adopte le ton d'un développeur qui raconte son stage à un collègue senior — concret, factuel. Le jury évalue si tu comprends ce que tu as fait, pas si tu récites.
Pose le CDC sur la table dès le début. Montre le guide HTML à la fin. Ces deux objets encadrent visuellement ton exposé.
Ne lis pas tes notes. Si tu dois les consulter, fais-le naturellement — pas comme si tu apprenais le texte en direct.
1 Contexte entreprise 2 min

"J'ai effectué mon stage de 2ème année chez Komidi, une association culturelle à Saint-Joseph, La Réunion. Komidi utilise TiBillet — une suite open source de billetterie développée par la coopérative Code Commun — pour gérer ses ventes de billets. Ma mission couvrait trois axes : mettre en place l'environnement technique, développer des fonctionnalités demandées, et résoudre les incidents rencontrés."

Geste : pose le CDC sur la table en disant "ma mission était formalisée dans ce document". Ne l'ouvre pas encore — juste le montrer suffit.
Transition : "Le travail était organisé en mode projet, avec des cahiers des charges et des issues numérotées…"
3 phrases max sur l'entreprise — le jury veut entendre la technique, pas l'histoire de Komidi.
2 C4 Travail en mode projet 1 min

"Avant de coder quoi que ce soit, nous avons rédigé deux cahiers des charges — un pour le front-end, un pour le back-end — qui analysaient les fonctionnalités existantes de TiBillet par rapport aux besoins de Komidi. Chaque tâche était classée par statut et niveau de difficulté. Les développements ont ensuite été suivis sous forme d'issues numérotées, chacune documentée dans un fichier séparé."

Ouvre le CDC et montre physiquement le tableau. Le jury voit immédiatement que c'est un vrai document de projet, pas une liste faite après coup.
Transition : "Je vais commencer par la mise en service technique, puis passer aux développements réalisés."
1 min seulement ici — ne rentre pas dans le détail du CDC. C'est un objet de cadrage, pas de démonstration.
3 C5 Installation et mise en service 3 min

"TiBillet est composé de trois moteurs indépendants qui tournent chacun dans leur propre conteneur Docker. Fedow s'occupe des transactions et de la fédération entre structures. Lespass gère la billetterie et les adhésions — c'est lui que Komidi utilise côté public. LaBoutik, lui, est la caisse cashless — il n'était pas utilisé dans ce stage. Ces trois moteurs ont besoin de se parler, et pour sécuriser ces échanges, TiBillet utilise des clés Fernet. C'est simple : les deux services partagent la même clé secrète dans leur fichier de configuration — si elles ne correspondent pas, les moteurs ne peuvent tout simplement pas communiquer. Une fois l'environnement monté, je vérifiais que tout fonctionnait via le shell Django."

L'image à garder en tête : trois boîtes Docker qui parlent entre elles — Fernet c'est le "mot de passe commun" qui autorise cette conversation.
Si le jury demande pourquoi Docker : "Chaque service tourne isolé avec ses propres dépendances — reproductible sur n'importe quelle machine, sans risque de conflit de versions."
Pas besoin de citer les commandes ici. L'objectif de cette partie c'est de montrer que tu comprends l'architecture, pas que tu récites une procédure.
4 C3 Développement de fonctionnalités 4 min

"J'ai développé plusieurs fonctionnalités directement dans le code de TiBillet. Je vais vous donner un exemple : le filtre des événements par date. Les utilisateurs ne pouvaient filtrer les spectacles que par catégorie — pas par date. J'ai modifié une fonction dans le fichier views.py pour prendre en compte la date choisie. Les dates sur la page sont devenues des liens cliquables. Et j'ai créé un nouveau fichier de template avec un panneau de filtres géré entièrement en CSS, sans JavaScript. Sur le même principe, j'ai réalisé les autres fonctionnalités."

Rythme en 3 temps : (1) le besoin de Komidi, (2) le fichier Python touché, (3) le résultat visible dans l'interface. Ça montre que tu suis le fil modèle → logique → affichage.
Transition naturelle : "Sur le même principe, j'ai aussi…" puis enchaîne vite sur les autres issues sans t'y attarder.
Pour les mentions légales : "La mise en ligne d'un service implique des obligations légales — j'ai rédigé les mentions légales conformes LCEN et RGPD."
Ne récite pas les 6 issues dans l'ordre. Un exemple bien expliqué vaut mieux que six noms de variables lâchés sans contexte.
5 C2 Incidents résolus 3 min

"J'ai eu deux types d'incidents distincts : un en production sur le VPS de Komidi, et un en développement sur mon environnement local. Sur le VPS, le site ne chargeait plus — j'ai diagnostiqué un Client mal configuré en base de données et corrigé directement via le shell Django. En développement, l'environnement Docker était instable dès le départ — les conteneurs redémarraient en boucle et bloquaient toute initialisation."

Pour les 5 problèmes Docker : présente la chaîne causale, pas la liste. "La vraie difficulté n'était pas la correction — c'était trouver la cause racine. J'ai dû croiser les logs Docker, l'état des branches Git et la version Docker avant de comprendre que LaBoutik saturait les ressources."
Formule clé : "J'ai identifié le symptôme, cherché la cause, appliqué une solution ciblée, vérifié le résultat." — c'est exactement la définition de C2 dans le référentiel.
Évite "j'ai cherché sur Google". Parle d'analyse des logs, lecture du code, tests successifs. Le jury veut une démarche, pas une anecdote.
6 Bilan et livrable 2 min

"Pour conclure ce stage, j'ai produit deux livrables : un guide d'installation HTML complet avec table des matières et section troubleshooting documentant les 5 problèmes résolus — et le site de billetterie de Komidi avec les fonctionnalités livrées. Ce stage m'a permis de toucher à l'ensemble de la stack d'un projet open source réel : de l'infrastructure Docker jusqu'aux templates Django, en passant par la correction d'incidents en production."

Montre le guide HTML : ouvre la section "Problèmes rencontrés" — les 5 cartes colorées avec les commandes Docker sont visuellement parlantes pour le jury.
Dernière phrase : "C'est ma première expérience de contribution à un projet open source sous licence AGPL-3.0 — un logiciel lauréat du programme France 2030 du Ministère de la Culture."
Ne dis pas "est-ce que vous avez des questions ?". Termine ta phrase, regarde le jury, attends. C'est plus professionnel.
★ Pendant l'entretien
Si tu ne sais pas répondre : "Je n'ai pas eu à manipuler cette partie directement, mais je peux vous expliquer comment je m'y prendrais pour trouver la réponse." — honnêteté + démarche = bonne impression.
Si le jury reformule : c'est souvent qu'il veut plus de précision, pas que tu aies répondu faux. Creuse le détail.
Évite : "en gros", "du coup", "c'est en fait". Parle lentement et distinctement.

1 Vue d'ensemble — compétences et timing

L'épreuve E5 dure 30 minutes — exposé (~15 min) + entretien (~15 min). Ce stage couvre 4 compétences sur 6, c'est l'un des dossiers les plus complets qu'un candidat BTS SIO peut présenter.

#PartieCompétenceDurée cible
1Contexte entreprise + mission formalisée—2 min
2Cahiers des charges — travail en mode projetC41 min
3Installation et mise en service (Docker, Debian)C53 min
4Développement de 6 fonctionnalités (issues #180–#277)C34 min
5Incidents résolus : Bug VPS + 5 problèmes DockerC23 min
6Bilan — livrable guide HTML—2 min
Posture globale pour les 15 minutes
Adopte le ton d'un développeur qui raconte son stage à un collègue senior — concret, factuel, sans jargon inutile. Le jury évalue si tu comprends ce que tu as fait, pas si tu récites.
Pose le Cahier Des Charges sur la table dès le début. Montre le guide HTML à la fin. Ces deux objets encadrent visuellement ton exposé.
Ne lis pas tes notes. Si tu dois les consulter, fais-le naturellement, comme si tu vérifiais un détail technique — pas comme si tu apprenais le texte en direct.

4 Partie 3 — C5 Installation et mise en service 3 min

Les 6 étapes de mise en service

#ActionTechnologie
1Installer le système d'exploitationDebian 13, LVM, sudo
2Isoler les services dans des conteneursDocker, docker-compose
3Versionner le code sourceGit, SSH, GitHub
4Sécuriser les communications inter-servicesClés Fernet + Django Secret Key
5Démarrer les conteneurs + initialiser les BDDdocker compose up -d → flush.sh
6Vérifier le fonctionnementfedow.can_fedow() → True

Ordre d'initialisation imposé : fedow_django d'abord, puis lespass_django. Lespass dépend de Fedow pour s'initialiser — inverser l'ordre provoque des erreurs.

# Vérification finale — preuve que le service est opérationnel
config = Configuration.get_solo()
# → <Configuration: Settings for Lespass>  ✓

fedow = FedowConfig.get_solo()
fedow.can_fedow()
# → True  ✓  Lespass communique avec Fedow
Comment aborder cette partie
"Pour mettre le service à disposition, j'ai installé l'environnement complet sur une machine vierge sous Debian — en 6 étapes. L'étape clé était la génération des clés de chiffrement : TiBillet utilise Fernet, un algorithme symétrique, pour sécuriser les échanges entre Fedow et Lespass. Ces clés doivent être identiques dans les deux fichiers .env — sinon les deux moteurs ne peuvent pas communiquer. À la fin, je vérifiais le bon fonctionnement via le shell Django : si fedow.can_fedow() retourne True, tout est opérationnel."
Ce que tu dois mettre en valeur : la compréhension de l'architecture (3 moteurs indépendants) et la rigueur de la vérification finale. Pas juste "j'ai lancé docker compose up".
Si le jury te demande pourquoi Docker : "Chaque service tourne dans un conteneur isolé avec ses propres dépendances — reproductible sur n'importe quelle machine, sans risque de conflit de versions."
Évite : la liste de commandes une par une. Parle de l'intention, pas de la syntaxe. La syntaxe, c'est pour l'entretien si on te demande.

5 Partie 4 — C3 Développement de fonctionnalités 4 min

#277 Affichage conditionnel de l'heure d'un événement
Champ show_time = BooleanField dans models.py → admin Django (list_display + fields dans EventAdmin) → conditionnel dans 2 templates. Migration requise : makemigrations + flush.sh.
#257 Affichages configurables de la billetterie
Champs show_price_note + price_note dans models.py — masquer ou personnaliser le bloc prix. Balise template {% event_dates event %} créée dans tibitags.py — gère 1 jour, même jour avec fin, multi-jours. Mis à jour dans 4 templates.
#191 Affichage de la date/heure de fin d'événement
Bloc conditionnel dans retrieve.html — si même jour : affiche seulement l'heure de fin ; si jours différents : affiche la date complète de fin.
#180 Description courte dans l'agenda
Bloc {% if event.short_description %} dans partial/list.html — affiche la description sous le titre, rien si vide.
Frontend Filtre multi-critères des événements par date
views.py : paramètre date dans federated_events_filter(). Templates : dates deviennent des liens ?date=YYYY-MM-DD. Nouveau fichier search2.html — panneau "Options avancées" toggle CSS.
Back-end Recherche multi-critères des billets (admin)
admin_tenant.py — ajout de first_name, last_name, email dans list_filter de TicketAdmin.
Contenu Mentions légales (LCEN + RGPD)
Rédaction des 9 articles : éditeur (Komidi, RNA W9R2002592), hébergeur (OVHcloud), solution (TiBillet AGPL-3.0 France 2030), propriété intellectuelle, RGPD (base légale, durée de conservation, droits CNIL), paiement SSL/TLS, cookies.
Comment aborder cette partie
"J'ai développé six fonctionnalités directement sur le code source Django de TiBillet — sur les quatre couches de la stack : les modèles, les vues, l'admin, et les templates. Par exemple, pour l'issue #277, j'ai ajouté un champ BooleanField show_time dans le modèle Event en Python, propagé ce champ dans l'interface d'administration Django, appliqué la migration via un flush, puis modifié deux templates HTML pour conditionner l'affichage."
Structure en 3 temps pour chaque issue : (1) le besoin, (2) le fichier modifié, (3) le résultat visible. Pas besoin de réciter toutes les issues — choisis #277 et #257 comme exemples, et cite les autres en passant.
Formule de transition entre les issues : "Sur le même principe, j'ai aussi…" — ça montre que tu maîtrises la logique, pas juste l'exécution mécanique.
Pour les mentions légales : "La mise en ligne d'un service implique aussi des obligations légales — j'ai rédigé les mentions légales conformes LCEN et RGPD, incluant les droits des utilisateurs et le traitement des données personnelles."
Évite : de lire la liste des 7 fonctionnalités une à une. Le jury s'endort. Prends 2–3 exemples concrets et dis "avec d'autres modifications similaires".

6 Partie 5 — C2 Incidents résolus 3 min

Incident 1 — Bug VPS (production)

Symptôme : le site de billetterie de Komidi ne chargeait plus — Django levait une exception dès la page d'accueil.

Diagnostic : un Client en base avait une mauvaise categorie pour son schema_name — Django ne pouvait pas résoudre le tenant.

Correction :

docker exec -it lespass_django sh -lc "
python manage.py shell -c \"
from Customers.models import Client
c = Client.objects.get(schema_name='ce13207cf99c412986c8d6fec1943d3d')
c.categorie = 'S'
c.save()
\""
# Site opérationnel immédiatement après  ✓

Incident 2 — Environnement Docker instable (développement)

CAUSE RACINE :
  LaBoutik inclus dans docker-compose.yml sans être nécessaire
↓
Surcharge mémoire/CPU → conteneurs redémarrent en boucle
↓
./flush.sh et rsp s'interrompent sans raison apparente
↓
Diagnostic difficile → vérifications manuelles (logs, git branch, docker --version)
↓
Solution : suppression LaBoutik + git stash → git pull
↓
Environnement stable — tous les scripts fonctionnels ✓
Comment aborder cette partie
"J'ai eu deux types d'incidents distincts : un en production sur le VPS de Komidi, et un en développement sur mon environnement local. Sur le VPS, le site ne chargeait plus — j'ai diagnostiqué un Client mal configuré en base de données et corrigé directement via le shell Django. En développement, l'environnement Docker était instable dès le départ — les conteneurs redémarraient en boucle et bloquaient toute initialisation."
Pour les 5 problèmes Docker : ne les liste pas un à un — présente la chaîne causale. "La vraie difficulté n'était pas la correction elle-même, c'était trouver la cause racine. J'ai dû croiser les logs Docker, l'état des branches Git et la version Docker avant de comprendre que LaBoutik saturait les ressources."
Formule clé pour montrer une démarche méthodique : "J'ai identifié le symptôme, cherché la cause, appliqué une solution ciblée, vérifié le résultat." C'est exactement la définition de C2 dans le référentiel.
Évite : "j'ai cherché sur Google". Même si c'est vrai, le jury veut entendre une démarche, pas une anecdote. Parle d'analyse des logs, de lecture du code, de tests successifs.

8 Questions probables du jury

❓ "C'est quoi Docker ?"

Outil de conteneurisation : chaque service (Fedow, Lespass, PostgreSQL) tourne dans un conteneur isolé avec ses propres dépendances. Avantage : reproductible sur n'importe quelle machine sans conflit de versions.

❓ "C'est quoi Django ?"

Framework web Python qui suit le pattern MVC (Model-View-Template). TiBillet l'utilise pour gérer la base de données (models), la logique métier (views) et l'affichage HTML (templates).

❓ "Pourquoi Fernet ?"

Algorithme de chiffrement symétrique. TiBillet l'utilise pour sécuriser les communications entre Fedow et Lespass — les deux fichiers .env partagent la même clé. Si les clés diffèrent, les deux moteurs ne peuvent pas communiquer.

❓ "C'est quoi un BooleanField Django ?"

Un champ de modèle qui stocke True ou False en base de données. Pour show_time et show_price_note, il permet à l'admin Komidi d'activer ou désactiver l'affichage d'une information sans modifier le code.

❓ "C'est quoi une migration Django ?"

Quand on ajoute un champ au modèle, Django doit mettre à jour la structure de la base de données. makemigrations génère le fichier de migration, et flush.sh l'applique ici en réinitialisant aussi les données.

❓ "C'est quoi un tenant Django ?"

Isolation multi-tenant : public (utilisateurs partagés), lespass (données billetterie), meta (futur agrégateur). Permet à une instance TiBillet de gérer plusieurs organisations sur la même BDD.

❓ "Vous avez contribué au code officiel de TiBillet ?"

Non — j'ai travaillé sur le fork de Komidi. Ma mission était d'adapter TiBillet aux besoins spécifiques de l'association. Une contribution upstream aurait nécessité des tests plus approfondis et une validation par la coopérative Code Commun.

❓ "Comment avez-vous diagnostiqué le Bug VPS ?"

Les logs Django montraient une exception au chargement liée au résolveur de tenant. En inspectant le modèle Client via le shell Django, j'ai trouvé que la categorie du schema_name était incorrecte — valeur attendue 'S', valeur trouvée différente. Correction directe via manage.py shell.

❓ "C'est quoi AGPL-3.0 ?"

Licence copyleft forte — toute modification doit être rendue publique sous la même licence, même pour un service en ligne (contrairement à la GPL). C'est ce qui garantit que TiBillet reste libre même si des organisations l'adaptent pour leur usage.

❓ "Pourquoi VSCodium et pas VS Code ?"

VSCodium est la version open source de VS Code sans télémétrie Microsoft — cohérent avec la philosophie open source et coopérative de TiBillet et de Komidi.

❓ "Les mentions légales, c'est du développement ?"

C'est de la production de contenu avec une contrainte légale forte — LCEN et RGPD. La mise en ligne d'un service implique ces obligations. C'est une réalisation à part entière qui contribue à la présence en ligne de l'organisation — compétence C3.

Comment gérer l'entretien
Si tu ne sais pas répondre : "Je n'ai pas eu à manipuler cette partie directement, mais je peux vous expliquer comment je m'y prendrais pour trouver la réponse." — honnêteté + démarche = bonne impression.
Si le jury reformule une question : c'est souvent qu'il veut plus de précision, pas que tu aies répondu faux. Creuse le détail.
Évite : "en gros", "du coup", "c'est en fait". Parle lentement et distinctement — l'hésitation sur le fond se ressent moins si la diction est claire.

10 Ce qui te différencie

1. Quatre compétences E5 sur un seul stage

C2 + C3 + C4 + C5 — avec preuves concrètes pour chacune. La majorité des candidats couvrent 2 compétences max par réalisation.

2. Toute la pile Django — pas juste des templates

models.py, views.py, admin.py, tibitags.py, templates — tu as travaillé sur toutes les couches MVC d'un projet open source réel sous AGPL-3.0, lauréat France 2030.

3. Deux incidents documentés — développement ET production

Bug VPS corrigé via Django shell + 5 problèmes Docker résolus avec chaîne causale reconstruite. C'est du diagnostic méthodique, pas du tâtonnement.

4. Issues numérotées et traçables

#180, #191, #257, #277 — chaque fonctionnalité documentée dans un fichier .docx dédié avec fichiers modifiés et numéros de ligne. Niveau de traçabilité professionnel.

5. Livrable autonome de qualité professionnelle

Le guide HTML peut servir à tout futur contributeur sans toi. Table des matières, blocs copiables, alertes contextuelles, section troubleshooting — document de production, pas un brouillon de cours.

Entraîne-toi à voix haute avec un chronomètre. 15 minutes max pour l'exposé. Si tu dépasses 13 min à l'entraînement, coupe dans la partie C3 (moins de détails sur chaque issue). Ne jamais rogner sur C2 — c'est ta partie la plus forte.
